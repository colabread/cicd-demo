# 前端代码 Review 规范

## 使用说明

本文档用于补充 ESLint 难以自动检查的团队规范，重点覆盖代码语义、可维护性、可读性、可访问性、工程设计和 Review 判断。ESLint 能稳定自动发现的问题，应优先沉淀到 ESLint 规则中；本文档不替代自动化检查。

PR 人工 Code Review 时，Reviewer 可根据规则编号直接提出修改建议，例如 `REACT-003`。如果某条规则不适用于当前场景，应在 PR 评论或代码注释中说明原因，避免后续反复讨论。

后续引入 AI Code Review 时，应将本文档作为审查上下文。AI Review 应优先指出明确违反规则、影响维护性或用户体验的问题，避免对存在合理例外的实现做机械化拦截。

## 维护约定

- 规则允许新增、删除和更新，但每次变更应说明原因。
- 每条规则应保持可判断、可举例、可追溯，避免只表达抽象偏好。
- 新增规则时应补充 `规则编号`、`规则`、`Review 关注点` 和 `例外`。
- 规则编号按分类递增，历史编号不建议复用。
- 当规则可以被稳定自动化检查时，应优先考虑迁移到 ESLint、类型检查或其他工程工具。

## HTML 规范

### HTML-001

**规则**：优先使用语义化标签，避免无意义的 `div` 或 `span` 堆叠。

**Review 关注点**：检查页面结构是否使用了合适的 `header`、`main`、`section`、`article`、`nav`、`aside`、`footer`、`button`、`a`、`form` 等标签；仅承担布局职责的容器应保持简洁。

**例外**：组件库或框架组件已封装语义，或当前节点确实只用于布局、动画、定位时，可以使用无语义容器。

### HTML-002

**规则**：表单、按钮和链接必须符合真实交互语义。

**Review 关注点**：触发动作使用 `button`，页面跳转或资源访问使用 `a`；表单字段应关联 `label`；按钮应声明合理的 `type`，避免表单内按钮默认提交。

**例外**：受第三方 SDK 或组件库限制时，可使用等价的无障碍属性补足语义。

### HTML-003

**规则**：图片、图标和交互控件应具备合理的可访问性文本。

**Review 关注点**：有信息含义的图片应提供准确 `alt`；纯装饰图片可使用空 `alt`；仅图标按钮应有 `aria-label` 或等价文本；SVG 图标应区分展示型和语义型。

**例外**：图片内容已被相邻文本完整表达时，可使用空 `alt` 避免重复朗读。

### HTML-004

**规则**：避免将布局职责、业务含义和样式类名混杂在同一命名中。

**Review 关注点**：类名应描述模块、组件或稳定语义，避免使用 `red-text`、`margin-top-20`、`left-box` 等临时视觉命名；结构层级应能反映页面或组件意图。

**例外**：原子化 CSS、工具类 CSS 或设计系统已有统一命名约定时，按团队既有约定执行。

### HTML-005

**规则**：外链、下载和嵌入内容需考虑安全与用户预期。

**Review 关注点**：新窗口打开的外链应补充 `rel="noreferrer"` 或同等安全处理；下载链接应让用户能预期文件行为；iframe、第三方嵌入和不可信 HTML 应评估权限、来源和注入风险。

**例外**：内部可信系统、已由框架统一处理安全属性，或产品明确要求保留来源信息时，可按具体场景处理。

## CSS 规范

### CSS-001

**规则**：样式命名应表达组件或语义，不表达临时视觉。

**Review 关注点**：检查类名是否能在视觉调整后继续成立；避免 `big-title`、`blue-card`、`right-area` 等随 UI 变化失效的命名。

**例外**：设计系统中的通用工具类、状态类和 token 类可使用约定化视觉命名。

### CSS-002

**规则**：避免高优先级选择器、深层嵌套和难复用覆盖。

**Review 关注点**：选择器应尽量短且局部化；避免滥用 `!important`、ID 选择器、过深后代选择器和依赖 DOM 细节的覆盖；新增样式不应要求调用方了解内部层级。

**例外**：覆盖第三方组件库、浏览器默认样式或遗留样式时，可在局部范围内使用更高优先级，并说明原因。

### CSS-003

**规则**：公共变量、颜色、间距、字号应优先沉淀为 token 或统一变量。

**Review 关注点**：重复出现的颜色、阴影、间距、字体、圆角、层级等应复用已有变量；新增视觉值应确认是否属于一次性场景。

**例外**：与内容强绑定的一次性视觉、临时实验或第三方品牌资源可直接使用具体值。

### CSS-004

**规则**：响应式样式应围绕布局断点与内容适配，而不是局部硬编码补丁。

**Review 关注点**：检查移动端、窄屏、宽屏是否通过合理布局规则适配；避免大量针对单个元素的魔法宽高、负边距和固定偏移；文本和交互控件不应在常见视口下溢出或重叠。

**例外**：修复浏览器兼容问题、适配特定嵌入容器或已知设备限制时，可使用局部补丁。

### CSS-005

**规则**：动画、阴影、层级和固定定位需有明确业务理由。

**Review 关注点**：动画不应影响操作效率或可访问性；阴影和层级应服务于信息结构；固定定位元素不应遮挡主要内容或交互；`z-index` 应控制在局部上下文内。

**例外**：品牌展示、关键反馈、必要的浮层和系统级导航可使用更强视觉表现。

## JavaScript 规范

### JS-001

**规则**：业务逻辑应保持小函数、单职责，避免隐式副作用。

**Review 关注点**：函数名称、输入和输出应清晰；数据转换、状态修改、请求调用、埋点等职责不要混在难以测试的大函数中；副作用应从函数签名或调用位置可感知。

**例外**：非常短的事件处理函数、脚本入口或一次性胶水代码可适度聚合逻辑。

### JS-002

**规则**：异步流程需处理 loading、失败、取消和重复触发等状态。

**Review 关注点**：检查接口请求、定时器、订阅、文件读写等异步逻辑是否处理错误；用户重复点击、组件卸载、请求乱序和超时是否有合理策略。

**例外**：非关键后台任务、无需用户感知的最佳努力逻辑，可记录失败并保持主流程可用。

### JS-003

**规则**：避免魔法值，关键常量需命名并靠近业务上下文。

**Review 关注点**：时间、数量、状态码、业务枚举、正则、阈值等不应散落在逻辑中；命名应表达业务含义，而不是只复述字面值。

**例外**：局部且语义显然的值，如数组下标 `0`、简单布尔判断、一次性样式微调值，可直接使用。

### JS-004

**规则**：错误处理应可定位、可恢复，不吞异常。

**Review 关注点**：`catch` 不应空置；错误日志应包含足够上下文；用户可恢复的错误应提供反馈；不可恢复错误应向上抛出或进入统一错误边界。

**例外**：明确允许失败的清理逻辑、兼容性探测或降级逻辑，可忽略错误但应保留简短说明。

### JS-005

**规则**：复杂判断需拆分为具名变量或函数。

**Review 关注点**：条件表达式包含多个业务概念、嵌套三元、长链式判断时，应拆分为语义化变量或函数；Reviewer 应能不展开全部上下文就理解判断意图。

**例外**：短小且与当前上下文强相关的判断可以内联。

## TypeScript 规范

### TS-001

**规则**：类型应表达业务约束，避免为了省事使用 `any` 或过宽类型。

**Review 关注点**：检查是否用 `any`、`object`、`Record<string, unknown>`、宽泛字符串等掩盖真实结构；能用精确联合类型、接口或泛型表达的场景，不应退化为无约束类型。

**例外**：接入未知第三方数据、迁移遗留代码或构建通用底层工具时，可临时使用宽类型，但应在边界处收窄。

### TS-002

**规则**：公开函数、组件 props 和接口返回值应优先显式建模。

**Review 关注点**：跨文件、跨模块、组件对外暴露的输入输出应有稳定类型名称；类型命名应表达领域概念；避免让调用方依赖复杂推断结果。

**例外**：局部私有函数、简单常量和测试辅助函数可依赖类型推断。

### TS-003

**规则**：联合类型和枚举式常量应覆盖状态边界。

**Review 关注点**：状态机、请求状态、业务类型、权限类型等应明确所有可能状态；新增状态时应检查渲染、转换、默认值和兜底逻辑是否同步更新。

**例外**：直接来自后端且范围不稳定的开放字符串，可保留兜底分支并在业务层做保护。

### TS-004

**规则**：类型转换需有来源说明或运行时保护。

**Review 关注点**：`as`、非空断言和强制类型转换应谨慎使用；来自接口、URL、localStorage、用户输入等运行时数据，应先校验再进入强类型业务逻辑。

**例外**：框架 API 类型缺失、第三方库声明不完整或测试构造数据时，可使用类型断言，但应控制在最小范围内。

### TS-005

**规则**：避免类型与实际运行数据脱节。

**Review 关注点**：接口字段可选性、默认值、空值、数组长度和后端返回结构应与真实数据一致；不要只通过修改类型消除报错而不处理运行时分支。

**例外**：接口契约已由后端、schema 或生成代码保证时，可信任生成类型。

## React 规范

### REACT-001

**规则**：组件职责应清晰，避免展示、状态、数据请求和业务编排过度混杂。

**Review 关注点**：组件应能用一句话描述职责；复杂页面可拆分为容器、展示组件和业务 hook；组件过长或包含多个独立业务区域时，应评估拆分。

**例外**：小型页面、原型验证或高度内聚的简单交互可保持在单组件内。

### REACT-002

**规则**：props 设计应稳定、语义化，避免透传过多不受控参数。

**Review 关注点**：props 应表达组件能力和业务语义；避免把整块对象或大量 HTML 属性无选择透传给内部节点；回调命名应表达触发时机和业务含义。

**例外**：基础组件、设计系统组件和受控封装组件可支持合理透传，但应明确透传边界。

### REACT-003

**规则**：state 应保持最小必要，不存可派生数据。

**Review 关注点**：能从 props、URL、缓存、查询结果或其他 state 推导的数据，不应重复存储；多个 state 同时变化时，应检查是否存在不同步风险。

**例外**：昂贵计算结果、用户编辑中的草稿态、动画中间态或需要冻结历史值的场景可单独存储。

### REACT-004

**规则**：Effects 只用于同步外部系统，避免把普通计算塞进 `useEffect`。

**Review 关注点**：`useEffect` 应主要用于请求、订阅、DOM、定时器、外部状态同步等副作用；派生数据应优先在渲染期间计算或使用 `useMemo`；Effect 依赖应反映真实使用值。

**例外**：兼容第三方库生命周期、处理非 React 状态源或与浏览器 API 同步时，可以使用 Effect。

### REACT-005

**规则**：列表、条件渲染、空状态、错误状态和加载状态需完整考虑。

**Review 关注点**：列表应使用稳定 key；异步数据区域应覆盖 loading、empty、error、success；条件渲染不应导致布局跳动、交互丢失或状态残留。

**例外**：纯静态内容、确定永远有数据的内部配置或上游已统一处理状态的组件，可省略局部状态分支。

### REACT-006

**规则**：可复用逻辑优先抽 hook，但不为少量重复过早抽象。

**Review 关注点**：多个组件共享的状态机、订阅、请求编排、表单逻辑可抽为 hook；只有两三行且语义不稳定的重复代码，不应为了形式复用增加抽象成本。

**例外**：为统一行为、埋点、权限、实验或基础设施能力建立的 hook，可在早期抽象。
